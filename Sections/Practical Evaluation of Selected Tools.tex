\section{Practical Evaluation of Selected Tools}
~\label{chap:practical-evaluation}
\subsection{SimpleRisk Core}

\subsubsection{Tool Overview}

SimpleRisk Core is a free, open-source Governance, Risk, and Compliance (GRC) platform
designed to give organisations a structured starting point for information security risk
management. The platform is web-based and GUI-driven, presenting all functionality through
a browser interface organised into clearly labelled sidebar modules. It does not require
command-line interaction during normal use; after initial deployment, all configuration,
risk entry, and reporting is performed through the web UI.

The tool is distributed as a pre-built virtual machine image, which simplifies deployment
in test and small-production environments. It can also be installed manually on a Linux
server with a standard LAMP stack. SimpleRisk Core is available at no cost, while
advanced features such as extended integrations, enhanced reporting, and additional
compliance frameworks are offered through paid add-on modules.

The platform centres on a \textit{risk register} where risks are submitted, assigned to
owners, scored using configurable scoring methodologies, and tracked through a mitigation
and review workflow. Supporting modules cover governance documentation, compliance
testing, asset management, and reporting. An artificial intelligence module is listed in
the interface, though its capabilities in the Core (free) tier are limited.

\subsubsection{Installation}

SimpleRisk supports four installation methods: a scripted installation, a virtual machine
image, a Docker-based deployment, and a manual installation on a Linux server running
the LAMP stack (Linux, Apache, MySQL, and PHP). The scripted option automates the
setup on a supported Linux distribution using a single command, while the Docker option
packages the application in a container suitable for both local and cloud environments.
The manual installation provides the most control but carries a warning in the official
documentation that it should only be used where the other methods are not viable, as
operating system updates can cause the instructions to become outdated.
 
For this evaluation, the VirtualBox virtual machine image was used. This option requires
no manual server configuration and is the most accessible path for users without a Linux
administration background. After importing the image into VirtualBox and starting the
VM, the IPv4 address was retrieved using \texttt{ip addr}. The web interface was then
accessible at \texttt{https://<IP>}, where an administrator account was created on
first login. The default VM credentials are publicly documented, so changing them
immediately after deployment is a necessary first step before any meaningful use.

\subsubsection{Platform Modules}
\label{sec:simplerisk-modules}
SimpleRisk Core organises its functionality into the following main modules, accessible
from the left-hand sidebar.

\paragraph{Governance}
The Governance module provides tools for defining and managing the control frameworks
and policy documentation that underpin a risk management programme. Users can import
or define control frameworks and map controls to risks assessed elsewhere in the platform.
Policy documents such as information security policies and procedures can be stored and
versioned within the tool, supporting traceability of governance decisions over time.
Exception handling is also available, allowing deviations from controls to be formally
recorded and tracked.

\paragraph{Risk Management}
The Risk Management module is the operational core of the platform. Risks are submitted
through a structured form and stored in a central risk register. Each risk entry can be
assigned an owner, linked to an asset, given a scoring value, and associated with a
mitigation plan and review schedule. The platform supports six scoring methodologies:
Classic, CVSS, DREAD, OWASP, Custom, and Contributing Risk. Risks progress through a
defined workflow from submission through mitigation planning to periodic management
review, providing a basic but functional level of process support and auditability.

\paragraph{Compliance}
The Compliance module supports structured testing of controls against selected frameworks.
Through integration with the Compliance Forge SCF (available for free upon registering
the SimpleRisk instance), users gain access to a large library of controls mapped to
multiple recognised frameworks. Compliance test results can be recorded and tracked,
offering a degree of audit readiness. The depth of compliance automation is limited in
the Core tier compared to commercial GRC platforms.

\paragraph{Asset Management}
The Asset Management module allows organisations to define and catalogue their information assets, which can then be referenced when submitting risks. This enables a basic level of asset-centric risk tracking. Assets can be added manually through a form, or discovered automatically using the built-in Automated Discovery feature, which scans a specified IP range and adds live hosts to the asset list. The module does not provide integration with configuration management databases (CMDBs), so assets outside the network scope, such
as cloud services, must still be entered manually.

\paragraph{Artificial Intelligence}
The Artificial Intelligence module consists of two pages: \textit{Recommendations} and
\textit{Document Installation}. The Recommendations page allows SimpleRisk to generate
risk recommendations by connecting to Anthropic's API, which requires an administrator
to supply a valid Anthropic API key. This means the feature incurs external usage costs
and is not self-contained within the Core tier. The Document Installation page offers
one-click installation of policy and procedure document templates, but this functionality
is locked behind the paid Artificial Intelligence Extra add-on. In practice, the AI
module provides no usable functionality out of the box in the Core tier and is better
understood as a commercial upsell point than a core feature.

\paragraph{Reporting}
The Reporting module provides a set of pre-built reports and dashboards covering the
current risk landscape, mitigation status, and compliance posture. Reports can be
exported to PDF or CSV format, which supports sharing findings with stakeholders and
satisfies basic documentation requirements. Visualisation is limited to simple charts
and summary tables within the dashboard; the tool does not offer advanced or interactive
risk visualisation such as heat maps or bowtie diagrams.

\paragraph{Configure}
The Configure module provides administrative and scoring configuration options for the
platform. The most relevant of these for risk analysis is the Configure Risk Formula
page, which contains three tabs.

The Risk Levels tab defines the score boundaries and colour representation for the four
default risk levels: Low, Medium, High, and Critical. Both the thresholds and colours
can be edited freely. However, the number of levels is fixed; it is not possible to add
or remove levels from this tab.

The Classic Risk Formula tab presents a fully configurable likelihood/impact matrix.
Unlike the Risk Levels tab, this view includes add and remove controls for both the
likelihood and impact axes, meaning the matrix dimensions can be expanded or reduced to
fit the organisation's needs. Labels and numeric values for each level can also be
edited individually.

The Contributing Risk Formula tab takes a different approach. Likelihood levels are
configurable with add and delete options, similar to the Classic formula. Impact is
calculated as a weighted sum across a set of named subjects, with Safety, SLA,
Financial, and Reputation provided by default, each assigned a weight of 0.25. Both
the subjects and their weights can be modified, and new subjects can be added, allowing
the impact calculation to reflect the organisation's specific priorities.

Throughout the platform, many pages include an ``About This Page'' button in the top
right corner that links directly to the relevant section of SimpleRisk's official
documentation. This is a thoughtful usability feature, particularly for users who are
new to GRC platforms, as it removes the need to search through unrelated documentation
pages. However, the feature is not fully implemented: during the evaluation, a number
of pages displayed an ``Under Construction'' notice when the button was clicked,
indicating that documentation had not yet been written for those pages. This limits
the practical value of the feature in the evaluated version.

\subsubsection{Standards and Framework Support}
SimpleRisk Core does not explicitly implement a single named framework such as ISO/IEC
27005 or the NIST Risk Management Framework as a structured workflow. Instead, it
provides a flexible risk register and scoring environment that can be adapted to support
the intent of these frameworks without enforcing their specific terminology or process
steps.

The platform's qualitative scoring approach, using likelihood and impact estimates to
derive a risk score, is broadly consistent with the risk assessment process described
in ISO/IEC 27005, though the tool does not guide users through the full context
establishment, asset valuation, and threat/vulnerability identification steps that the
standard prescribes. The risk register and mitigation tracking workflow can be used to
support the risk treatment and monitoring phases common to both ISO/IEC 27005 and the
NIST RMF, but the tool does not enforce or document compliance with those frameworks
in a verifiable way.

OCTAVE and FAIR are not directly supported. The platform lacks built-in threat catalogues
or the structured asset-profile approach central to OCTAVE, and it does not offer the
probabilistic, financially-quantified risk modelling that FAIR requires. EBIOS Risk
Manager, which structures risk analysis around feared events and attack scenarios, is
similarly not reflected in the platform's workflow or terminology.

The strongest area of standards alignment is compliance framework mapping via the
Compliance Forge SCF, which covers a broad range of recognised frameworks including
NIST SP 800-171, CIS Critical Security Controls, PCI DSS, and HIPAA. This makes
SimpleRisk Core more suitable as a compliance tracking tool than as a framework-driven
risk analysis platform.

In summary, SimpleRisk Core is best characterised as a flexible, framework-agnostic
risk register with compliance support. Its alignment with established risk management
standards depends significantly on how the organisation configures and uses the tool,
rather than being enforced by the platform itself.

\paragraph{Asset Registration.}
The eight assets identified in Table~\ref{tab:nordkonsult-assets} were registered in the
platform via \texttt{Asset Management > Manage Assets}. The Add Asset form contains the following fields:
 
\begin{itemize}
    \item \textbf{Asset Name} (required): the name of the asset.
    \item \textbf{Asset Valuation} (required): a dropdown menu with predefined monetary
          ranges from \$0 to \$1,000,000 in increments of \$100,000.
    \item \textbf{IP Address}: an optional field for network-based assets.
    \item \textbf{Site/Location}: a dropdown with predefined location options,
          configurable via \texttt{Configure > Add or Remove Values}.
    \item \textbf{Team}: a dropdown with predefined team options, also configurable
          via \texttt{Configure > Add or Remove Values}.
    \item \textbf{Asset Details}: a rich text field for additional descriptive

    \item \textbf{Mapped Controls}: controls from defined frameworks can be linked
          directly to the asset at registration time.
    \item \textbf{Tags}: free-text tags for grouping or filtering assets.
    \item \textbf{Associated Risks}: existing risks can be linked to the asset
          at registration time.
\end{itemize}
 
For the NordKonsult scenario, only the mandatory fields and the most relevant optional

\$400,001 to \$500,000; and low-criticality assets (A-05, A-07) were assigned

\paragraph{Control Definition.}
The proposed controls from Table~\ref{tab:nordkonsult-controls} were entered in the
Governance module under \texttt{Governance > Define Control Frameworks}. A custom
framework named ``NordKonsult Controls'' was created, and each proposed control was
added as a control entry within this framework. The Add Control form contains the
following fields:
 
\begin{itemize}
    \item \textbf{Control Short Name} (required): a concise label for the control
          (e.g., ``Enable MFA on Microsoft 365'').
    \item \textbf{Control Long Name}: an optional expanded name for the control.
    \item \textbf{Control Description}: a rich text field for describing the control's
          purpose and scope.
    \item \textbf{Supplemental Guidance}: an additional rich text field for
          implementation notes or references.
    \item \textbf{Control Owner}: a dropdown for assigning responsibility.
    \item \textbf{Control Class}: Technical, Operational, or Management.
    \item \textbf{Control Phase}: a dropdown covering both traditional categories
          (Physical, Procedural, Technical, Legal and Regulatory or Compliance) and
          NIST CSF function names (Govern, Protect, Identify, Detect, Recover, Respond).
    \item \textbf{Control Priority}: P0 through P3.
    \item \textbf{Current Control Maturity} and \textbf{Desired Control Maturity}:
          both use a six-level scale of Not Performed, Performed, Documented, Managed,
          Reviewed, and Optimizing.
    \item \textbf{Control Status}: Pass or Fail. This field is not configurable.
    \item \textbf{Mitigation Percent}: a numeric value representing the percentage by
          which this control reduces the inherent risk score when linked to a mitigation.
          When multiple controls are linked to a single mitigation, only the highest
          mitigation percentage among them is applied to calculate the residual risk
          score.
    \item \textbf{Mapped Control Frameworks}: allows the control to be linked to one
          or more defined frameworks. Each mapping requires a control number, which
          serves as a cataloguing identifier within that framework, and an optional
          reference text.
    \item \textbf{Mapped Assets}: allows the control to be associated with specific
          assets.
\end{itemize}
 
For the NordKonsult scenario, each control was given a short name, a brief description,
and an owner. Current Control Maturity was set to ``Not Performed'' and Control Status
was set to ``Fail'' to reflect that none of the proposed controls were in place at the
start of the scenario. Each control was mapped to the ``NordKonsult Controls'' framework
via the Mapped Control Frameworks field, using the control IDs defined in
Table~\ref{tab:nordkonsult-controls} (e.g., C-01a, C-01b) as the control number.
Reference Text was left blank as the NordKonsult Controls framework has no external
reference document.
 
SimpleRisk also offers the Compliance Forge Secure Controls Framework (SCF), which provides a pre-built library of 875 controls mapped to 148 frameworks. While this library was examined during the evaluation, the NordKonsult scenario used custom controls to better test the tool's flexibility in handling organization-specific risk treatments.

\textit{Observation:} The control form in SimpleRisk is more comprehensive than it
initially appears. The inclusion of both Current and Desired Control Maturity using a
six-level scale, which resembles a capability maturity model, allows organisations to
track not just whether a control exists, but how well it is implemented and where they
want it to be. This is a useful feature that goes beyond the simple pass/fail status and
supports more nuanced governance reporting. The Mitigation Percent field adds a further
layer of analytical value, as it enables residual risk scores to be calculated based on
the effectiveness of linked controls, though the constraint that only the single highest
percentage is applied may not reflect real-world scenarios where multiple complementary
controls collectively reduce risk. The Control Phase field is also notable, as its
inclusion of NIST CSF function names (Govern, Protect, Identify, Detect, Recover,

the platform does not provide direct mapping to Norwegian-specific frameworks such as
NSM's \textit{Grunnprinsipper for IKT-sikkerhet}, which would need to be created

alignment with international standards, but its breadth may be overwhelming for a small
organisation like NordKonsult. It should also be noted that the official SimpleRisk
documentation does not always reflect the current state of the platform; for example,
the documentation describes Supplemental Guidance as a file upload field, while the
platform itself presents it as a rich text editor. This discrepancy reinforces the
importance of direct hands-on evaluation over reliance on documentation alone.

\paragraph{Risk Submission.}
The eight risks from Table~\ref{tab:nordkonsult-risks} were entered via
\texttt{Risk Management > Submit Risk}. The Submit Risk form contains the following
fields:
 
\begin{itemize}
    \item \textbf{Subject}: a concise description of the risk.
    \item \textbf{Risk Mapping}: maps the risk against one of 32 predefined high-level
          risk categories, which can be used to group and report on risks at a management
          level. The available categories can be modified via the Risk and Threat Catalog
          in the Configure menu.
    \item \textbf{Threat Mapping}: links the risk to a defined threat source. This field
          is present in the platform but is not documented in the official Risk Submission
          documentation, which is a further example of a discrepancy between the platform
          and its documentation.
    \item \textbf{Category}: a dropdown for classifying the risk by threat type,
          configurable via the Add and Remove Values page in the Configure menu.
    \item \textbf{Site/Location}: the location associated with the risk, also
          configurable via the Add and Remove Values page.
    \item \textbf{External Reference ID}: an optional identifier for cross-referencing
          with external systems or documentation. Notably, entering a CVE identifier
          triggers automatic population of the risk details from the National
          Vulnerability Database, provided the instance has internet access.    \item \textbf{Control Regulation}: links the risk to a relevant regulation or
          compliance requirement.
    \item \textbf{Control Number}: an optional cataloguing identifier.
    \item \textbf{Affected Assets}: links the risk to one or more registered assets.
    \item \textbf{Technology}: the technology associated with the risk.
    \item \textbf{Team} and \textbf{Additional Stakeholders}: the teams or individuals
          with an interest in the risk.
    \item \textbf{Owner} and \textbf{Owner's Manager}: the person responsible for
          managing the risk and their manager.
    \item \textbf{Risk Source}: the origin of the risk (e.g., internal, external).
    \item \textbf{Risk Scoring Method}: the scoring methodology to apply. Set to
          Classic for all NordKonsult risks.
    \item \textbf{Current Likelihood} and \textbf{Current Impact}: the likelihood and
          impact values used to calculate the risk score.
    \item \textbf{Risk Assessment}: a rich text field for documenting the assessment
          rationale, including threat and vulnerability context.
    \item \textbf{Additional Notes}: a rich text field for supplementary information.
    \item \textbf{Supporting Documentation}: a file upload field for attaching
          relevant evidence or reference documents (max 5 MB).
    \item \textbf{Tags}: free-text tags for filtering and grouping.
\end{itemize}
 
For the NordKonsult scenario, the Subject, Category, Affected Assets, Owner, Risk
Scoring Method, Current Likelihood, Current Impact, and Risk Assessment fields were
populated for each risk. The Risk Assessment field was used to document the underlying
vulnerability and potential consequences of each risk. To illustrate the process, the
following provides detail on how three representative risks were submitted.
 
\subparagraph{R-01: Phishing Attack.}
This risk was assigned a likelihood of 4 (Likely) and an impact of 4 (Major), resulting
in a risk score of 16 (Critical). The affected assets were set to Microsoft 365 Tenant
(A-01), Client Project Data (A-02), and Employee Laptops (A-04). The Risk Assessment
field noted that employees may fall victim to phishing emails due to the absence of
recurring security awareness training, and that multi-factor authentication is not
enforced across all accounts. A successful phishing attack could result in unauthorised
access to email, client data, and internal communications.
 
\subparagraph{R-02: Ransomware Infection.}
This risk was assigned a likelihood of 3 (Possible) and an impact of 5 (Critical),
resulting in a risk score of 15 (High). The affected assets were Employee Laptops (A-04)
and On-Premise NAS (A-05). The Risk Assessment field highlighted the irregular backup
routines and absence of endpoint detection and response (EDR) software as the primary
vulnerabilities enabling this threat.
 
\subparagraph{R-04: Client Data Breach.}
This risk was assigned a likelihood of 3 (Possible) and an impact of 4 (Major),
resulting in a risk score of 12 (High). The affected assets were Client Project Data
(A-02) and CRM System (A-03). The Risk Assessment field noted overly broad file sharing
permissions in SharePoint and the absence of a data classification policy as key
contributing factors.

\textit{Observation:} The risk submission form in SimpleRisk is comprehensive and
supports a reasonable level of structured documentation. The presence of dedicated
Threat Mapping and Risk Mapping fields indicates that the platform does support linking
risks to defined threat sources and to predefined high-level risk categories, which goes
beyond simple free-text entry. However, in practice the Threat Mapping field depends on
threat sources being pre-defined in the system; without a built-in threat catalogue, the
analyst must populate this data manually before it becomes useful. The CVE auto-population
feature via the External Reference ID field is a notable practical capability, as it
reduces manual effort when documenting technically known vulnerabilities. The separation
of Risk Assessment and Additional Notes into two distinct rich text fields encourages
more thorough documentation, though neither field enforces a specific structure, meaning
documentation quality still depends on the analyst. It is also worth noting that the
Threat Mapping field is present in the platform but absent from the official
documentation, which is consistent with the pattern of discrepancies between the
platform and its documentation observed elsewhere in this evaluation. The Classic scoring
method provides a single likelihood and impact value per risk with no support for
uncertainty ranges or confidence levels, which would be insufficient for organisations
seeking quantitative risk analysis such as those following the FAIR methodology.

\paragraph{Mitigation Planning.}
For each submitted risk, a mitigation plan was defined via
\texttt{Risk Management > Plan Mitigations}. The mitigation form contains the following
fields:
 
\begin{itemize}
    \item \textbf{Planned Mitigation Date}: the target date for mitigation
          implementation.
    \item \textbf{Planning Strategy}: the chosen risk treatment approach, selected
          from Accept, Mitigate, Research, Transfer, and Watch. These options correspond
          directly to the risk treatment strategies defined in ISO/IEC 27005 and the
          NIST RMF.
    \item \textbf{Mitigation Effort}: an estimate of implementation effort using a
          five-level scale of Trivial, Minor, Considerable, Significant, and Exceptional.
    \item \textbf{Mitigation Cost}: a monetary range dropdown using the same scale
          as the Asset Valuation field.
    \item \textbf{Mitigation Owner}: the user responsible for executing the mitigation.
    \item \textbf{Mitigation Team}: the team responsible for implementation.
    \item \textbf{Mitigation Percent}: the percentage by which the mitigation reduces
          the inherent risk score to produce the residual risk score.
    \item \textbf{Mitigation Controls}: links to one or more controls defined in the
          Governance module.
    \item \textbf{Current Solution}: a rich text field for describing any existing
          measures already in place.
    \item \textbf{Security Requirements}: a rich text field for documenting the
          requirements the mitigation must satisfy.
    \item \textbf{Security Recommendations}: a rich text field for implementation
          guidance and recommended actions.
    \item \textbf{Supporting Documentation}: a file upload field for attaching
          relevant evidence or reference documents.
\end{itemize}

\paragraph{Summary.}
SimpleRisk Core provides a solid foundation for organisations establishing a structured
risk management programme. Its qualitative risk scoring, configurable workflows, and
compliance audit lifecycle cover the core requirements of the Fase 3 evaluation criteria,
with the Planning Strategy options showing notable alignment with ISO/IEC 27005 and NIST
RMF treatment strategies. The platform's strongest areas are its risk register workflow,
control and governance management, and the breadth of its compliance framework support
via the Compliance Forge SCF.
 
The tool's main limitations in the Core tier are the absence of built-in threat
catalogues, the lack of native integrations with SIEM, CMDB, or ticketing systems, and
the restriction of automation features such as audit notifications behind paid add-ons.
Quantitative risk analysis in the style of FAIR is not supported. Visualisation is
limited to basic pie charts, with no risk matrix or trend analysis available.
 
The evaluation also identified a recurring pattern of discrepancies between the
platform's official documentation and its actual behaviour, including fields that behave
differently than documented and features present in the platform that are absent from
the documentation entirely. The per-page documentation feature, while well-intentioned,
is only partially implemented in the evaluated version. These observations reinforce the
value of hands-on evaluation over documentation review alone, and are themselves
relevant findings for the comparative analysis in Chapter~\ref{chap:comparative}.

\subsection{OpenRMF OSS} % TODO: Add screenshots throughout
\subsubsection{Tool Overview}
 
OpenRMF OSS is a free, open-source web application designed to support the
implementation of the NIST Risk Management Framework (RMF) in U.S. Department of
Defense (DoD) environments. The tool addresses a specific and well-defined problem:
the management of DISA STIG checklists, SCAP scan results, and Nessus/ACAS patch
data is traditionally handled through a combination of the DISA STIG Viewer, manual
spreadsheets, and email. OpenRMF OSS replaces this workflow with a centralised,
browser-based interface that consolidates assessment data across systems and automates
the generation of key RMF artefacts.
 
The application is built on a microservices architecture using Docker, MongoDB, and
the NATS messaging system, with authentication and authorisation handled through
Keycloak using OpenID Connect and role-based access control. This design allows the
tool to be deployed locally, on a server, in a virtualised environment, or in the
cloud. However, the multi-component nature of the stack means that initial setup
requires a meaningful level of technical expertise compared to tools distributed
as a single VM image.
 
OpenRMF OSS is explicitly scoped to the DoD RMF process and the associated US
federal compliance ecosystem. It is not designed as a general-purpose information
security risk management tool and does not support the broader range of risk
management methodologies and organisational contexts relevant to European or
Norwegian environments. This scope should be understood as a defining
characteristic of the tool rather than a limitation in isolation.

\subsubsection{Installation}

OpenRMF OSS is distributed as a Docker Compose stack rather than as a
pre-built virtual machine image. The evaluated instance was deployed using the
official Docker Compose stack from the v1.14 release archive, downloaded from
the project's GitHub repository. The stack was hosted on a dedicated Windows
laptop running Docker Desktop with WSL2, and accessed remotely via browser
from a separate machine on the same network. The host machine's network IP
was configured in the \texttt{.env} file prior to launching the stack, and
Keycloak was initialised using the bundled Linux setup script executed through
a WSL2 terminal. The full stack, including all dependencies, was brought up
with a single \texttt{docker compose up -d} command.

\textit{Observation:} While the final \texttt{docker compose up -d} command is
simple, the deployment has markedly more prerequisites than the VM-image
distribution used by SimpleRisk Core. The evaluator is expected to be
comfortable with Docker, WSL2, and Keycloak concepts before the tool can be
used at all; user administration is delegated entirely to Keycloak's admin
console rather than being handled natively in OpenRMF, meaning an
administrator must learn two systems instead of one. For organisations
without in-house technical operations capability, this deployment model is a
meaningful barrier, and the absence of a pre-built VM image or hosted demo
excludes the tool from the same accessibility tier as SimpleRisk Core.

\subsubsection{Platform Modules}
\label{sec:openrmf-modules}

OpenRMF OSS organises its functionality into eight main pages, accessible
from the left-hand sidebar. Unlike SimpleRisk Core, the platform does not
require a separate scenario walkthrough to illustrate its workflow; the
checklist-centric nature of the tool means that module behaviour is best
described through a single worked example. For this purpose, the Microsoft
Windows 11 STIG (Version 2, Release 7) was uploaded to a test system package
and used to explore each module.

\paragraph{Dashboard.}
The Dashboard displays aggregate counts at the top (System Packages,
Checklists, and Templates) alongside two interactive widgets: one for Nessus
and ACAS patch-scan data and one for SCAP scans and checklist findings. Each
widget requires a System Package to be selected from a dropdown before any
content appears. After uploading the Windows 11 STIG to the test system
package, the SCAP Scans and Checklists widget populated with High, Medium,
and Low vulnerability tiles reflecting the current state of the uploaded
checklist.

\paragraph{System Packages.}
A System Package represents an ATO boundary — the unit of compliance
scrutiny in the NIST RMF process. Creating a package requires only a title
and description, with an optional Nessus/ACAS file upload at creation time.
Once created, the System Package page provides a central workspace that
links to all associated checklists, displays a system-level status chart,
summarises patch vulnerabilities, and exposes buttons for generating a Test
Plan and a Plan of Action and Milestones (POA\&M). This design reinforces
the tool's orientation around the ATO process: the system package is the
primary organisational container, not the individual asset.

\paragraph{Test Plan Summary and POA\&M Generation.}
The System Package page exposes two one-click report generators that produce
Microsoft Excel deliverables intended to serve as formal RMF artefacts.
The Test Plan Summary is a lightweight pre-assessment document: for the
Windows~11 test package, it produced a six-row file containing a header block
and a single-row summary table counting only the currently Open findings
(1~CAT~I, 3~CAT~II, 0~CAT~III, 4~total). The file's usefulness scales with the
number of checklists and hosts in the System Package; against a single test
checklist its content is minimal.

The POA\&M export is substantially more substantive. The generated file
reproduces the layout of the DoD Plan of Action and Milestones template,
with a 22-column finding table covering vulnerability description, security
control acronym, source STIG, status, severity, and fields for remediation
planning including Scheduled Completion Date, Resources Required, Milestones,
and Residual Risk Level. For the Windows~11 test package the export contained
254 finding rows with NIST~800-53 control mappings inlined per row
(for example, \texttt{CM-6 b} and \texttt{SC-28 (1)}), drawn directly from
the CCI data carried in the original DISA content.

\textit{Observation:} A subtle but important design decision is reflected in
how the POA\&M selects rows. The test checklist had 258 rules; four were
marked Open, three as Not a Finding, and one as Not Applicable during the
evaluation, leaving 250 findings in the default Not Reviewed state. The
POA\&M contained 254 rows, meaning all Open \textit{and} all Not Reviewed
findings were included with status ``Ongoing''. The Test Plan Summary, by
contrast, counted only the four Open findings. This asymmetry reflects
RMF practice, where unreviewed items are treated as outstanding work, but
it is worth understanding before the POA\&M is shared with stakeholders;
the document is better read as ``remaining work to be cleared'' than as
``confirmed open findings''. The column structure is also specific to US
federal compliance, with fields such as \textit{AP Acronym}, \textit{OMB
Project ID}, and \textit{DoD Component} that have no equivalent in
Norwegian or European regulatory contexts.

\paragraph{Checklists.}
Checklists are the operational core of OpenRMF and are uploaded either as
DISA STIG \texttt{.CKL} files or as SCAP scan \texttt{.XML} files, with up
to ten files per upload. Once processed, a checklist is presented through a
layout combining asset metadata, a four-status summary table (Open, Not a
Finding, Not Applicable, Not Reviewed) broken down by CAT~I, CAT~II, and
CAT~III severity, a vulnerability filter, and three linked panels:
Vulnerability IDs (the full list of rules), Vulnerability Details (rule
description, Check Content, Fix Text, and CCI-to-NIST 800-53 mappings), and
Status \& Comments (the editing interface).

The uploaded Windows 11 STIG contained 258 rules, automatically categorised
on upload as 28~CAT~I, 213~CAT~II, and 17~CAT~III, with every rule initially
set to \textit{Not Reviewed}. Marking a finding with a different status
(Open, Not a Finding, or Not Applicable) updates both the colour-coded
vulnerability list and all three summary panels immediately, without a page
reload. Each finding also exposes NIST SP 800-53 control mappings (Rev.~4
and Rev.~5) directly in the detail pane, along with the corresponding CCI
references from the underlying DISA content.

\paragraph{Compliance.}
The Compliance module generates a NIST 800-53 control coverage view from
all checklists within a selected System Package. The user selects a System
Package, sets a System Impact Level (Low, Moderate, or High) corresponding
to the FIPS~199 categorisation levels, optionally flags the system as
containing PII, and clicks Generate. For the Windows~11 test package, the
summary returned 16 NIST 800-53 control families (AC, AT, AU, CA, CM, CP,
IA, IR, MA, MP, PE, PL, PM, PS, RA, SA, SC) with each family colour-coded
to indicate whether any mapped findings were Open, Not Reviewed, or fully
satisfied. The Details view expanded this into a table of 191 individual
controls, filterable by family. The full listing can be exported to
Microsoft Excel, producing a file with the same columns (Control, Title,
Checklist, Status) and colour coding as the on-screen view.

\paragraph{Templates.}
OpenRMF OSS ships with 629 DISA-provided templates representing the public
portion of the STIG library. These are available as starting points for new
checklists without requiring manual download from the DISA website.
Organisations may also upload their own templates via the Upload page for
reuse across future system packages.

\paragraph{Reports.}
The Reports module offers twelve predefined reports, including a System
Package Pie Chart, Host Vulnerability Report, Checklist Dashboard, and RMF
controls listing. Reports are generated on demand and displayed inline. The
Pie Chart report, tested against the Win11 test package, produced a
distribution of findings by category and status; the Host Vulnerability
Report supported partial Vuln ID search (for example, \texttt{V-2533})
and returned a table of matching findings with host, severity, status, and
type columns.

\paragraph{Upload.}
The Upload module is split into two workflows: one for uploading
\texttt{.CKL} or SCAP \texttt{.XML} files as checklists against an existing
System Package, and one for uploading \texttt{.CKL} files as reusable
templates. Each workflow accepts up to ten files per upload, and the
inline help text explicitly notes that DISA SCAP, OpenSCAP, and Nessus SCAP
output formats are all supported.

\paragraph{Audit, Metrics, and Users.}
The Audit page logs every configuration and data change across the platform.
For the Windows~11 test session, ten audit entries were generated: one for
system package creation, one for checklist upload, and one per vulnerability
status change. Each entry records a timestamp, originating API component
(Save API, Upload API), action description, username, and user ID, and can
be expanded to reveal the full HTTP method and URL, the affected object
IDs, and a unique audit identifier. The audit page supports keyword search
and column sorting but no structured filtering.

The Metrics page redirects to a Grafana interface. In the default
deployment, logging in to Grafana was not possible without further
configuration, suggesting that the integration requires additional Keycloak
or Grafana setup beyond the standard installation. The Users page likewise
redirects to Keycloak's administrative console, confirming that user and
role management is handled externally to the OpenRMF application itself.

\textit{Observation:} OpenRMF's workflow is fundamentally different from
SimpleRisk's. Where SimpleRisk treats risks as free-form register entries
scored by the analyst, OpenRMF ingests structured DISA content and derives
findings, severities, and control mappings automatically from that content.
This shifts the analyst's work from risk authoring to finding triage: the
rules are already defined, categorised, and mapped; the analyst's job is to
determine whether each rule is met in the target system. The immediacy of
the UI reflects this — status changes propagate across the summary tables,
severity breakdowns, and dashboards without page reloads, encouraging a
fast triage cadence across a large rule set.

\textit{Observation:} The Compliance Summary view exposes a subtle but
important characteristic of STIG-to-NIST mapping: a single Open finding can
mark multiple control families as non-compliant because individual STIG
rules typically map to several CCIs, each of which maps to a NIST control.
In the Windows~11 test, four Open findings caused eight control families
to be flagged at the summary level. This means the compliance view should
be read as ``at least one finding in this family requires attention''
rather than as a direct count of open issues, which is a distinction worth
understanding before presenting the output to non-technical stakeholders.

\subsubsection{Standards and Framework Support}
OpenRMF OSS is the most narrowly scoped tool evaluated in this thesis in terms of
framework alignment. It is purpose-built for the NIST Risk Management Framework and
the associated US Department of Defense compliance ecosystem, and this alignment is
direct rather than aspirational. The tool's entire workflow, from data input through
to output artefacts, is structured around NIST SP 800-53 controls, DISA STIGs, and
the ATO (Authority to Operate) process.
 
This means that in the single area it targets, NIST RMF support, OpenRMF OSS
outperforms general-purpose tools such as SimpleRisk Core. Compliance reporting
against NIST 800-53 is automated rather than manual, POA\&M and RAR generation
require a single click rather than analyst effort, and the STIG and SCAP data inputs
are natively understood by the tool rather than requiring manual transcription.
 
However, outside of this context, the tool offers no meaningful support for any of
the other frameworks relevant to this evaluation. ISO/IEC 27005, OCTAVE, FAIR, and
EBIOS are not referenced in the tool's design, documentation, or outputs. The
qualitative risk assessment processes described by these frameworks, involving asset
identification, threat and vulnerability analysis, and risk treatment planning, are
not supported. Risk in OpenRMF OSS is expressed through STIG finding categories
(Category I, II, III) rather than through a likelihood/impact model, making the
tool's risk representation incompatible with the evaluation criteria used for the
other tools in this thesis.
 
For organisations operating outside the US DoD context, including Norwegian
enterprises subject to NSM guidance, ISO/IEC 27001, or GDPR, OpenRMF OSS is not
a viable risk management tool. Its inclusion in this thesis is valuable precisely
because it illustrates the contrast between a tool designed for a specific
compliance-driven framework and the more general-purpose tools that form the bulk
of the evaluation.

\subsubsection{Summary}

OpenRMF OSS delivers a focused, technically mature implementation of the
NIST Risk Management Framework for organisations operating within the DISA
STIG and NIST SP 800-53 ecosystem. Within this narrow scope, it performs
work that general-purpose GRC platforms cannot: STIG checklists are ingested
directly from DISA content, findings are automatically categorised by
severity, and NIST 800-53 control coverage is derived from CCI mappings
already embedded in the source material. The UI supports rapid finding
triage with live-updating summaries, and the Compliance and Reports modules
generate audit-ready Excel exports without analyst intervention. The audit
log is comprehensive, capturing every configuration and data change with
sufficient detail to support external review.

The tool's limitations follow directly from its scope. Risk is expressed
only through STIG finding categories, with no support for likelihood/impact
scoring, quantitative modelling, or qualitative risk authoring. Asset
management is absent: the organisational unit is the System Package (an ATO
boundary), not the individual asset. Frameworks other than NIST RMF
(ISO/IEC~27005, FAIR, OCTAVE, EBIOS) are not represented in the tool's
workflow or outputs. Deployment is gated by familiarity with Docker Compose,
WSL2, and Keycloak, and the Metrics integration did not function in the
default deployment used for this evaluation.

For organisations operating outside the US federal compliance context —
including Norwegian public and private sector organisations subject to NSM
guidance, ISO/IEC~27001, or GDPR — OpenRMF OSS is not a viable
general-purpose risk management tool. Its inclusion in this thesis is
valuable as a point of contrast: it demonstrates what a compliance-driven,
framework-specific tool can achieve in its target domain, and by the same
token illustrates the trade-off that specialisation imposes on broader
applicability.


\subsection{MONARC}

\subsubsection{Tool Overview}
MONARC (Optimized Risk Analysis Method) is an open-source risk analysis software that was originally developed by CASES (Cyberworld Awareness and Security Enhancement Services), a Luxembourg government cybersecurity initiative, and now maintained by the National Cybersecurity Competence Center. MONARC was primarily created to allow SMEs (Small and Medium-sized Enterprises) to benefit from a well structured risk analysis platform without requiring a significant financial investment or deep expertise in risk management 

The tool implements ISO/IEC 27005 managing information related risks based on the CIA triad criteria: Confidentiality, Integrity, and Availability. Risks are calculated based on three key factors: threat, vulnerability, and impact on the organization.

The method is structured into four phases, inspired by ISO/IEC 27005: 
\begin{enumerate}
    \item Context establishment
    \item Context modeling
    \item Evaluation and treatment of risks
    \item Implementation and monitoring
\end{enumerate}
The key advantage of MONARC is that it reuses risk analyzes from similar organizations, since businesses in the same field often face the same threats and vulnerabilities, meaning that risk assessments do not have to be rebuilt from scratch.

In general, MONARC is best suited for organizations that seek a structured, repeatable, and standards aligned approach to information security risk management. Its reusable risk models and low knowledge threshold make it especially relevant for SMEs, and multi-department organizations that want to conduct consistent risk assesments without building everything from the ground up.

\subsubsection{Installation}
MONARC can be installed in various ways. For our evaluated case, we decided to go with a VM image due to its simplicity, as it does not require a manual server configuration and is easier to use and maintain for users without a Linux administration background. The VM image was imported into Virtualbox, and after powering up the VM, the web interface that MONARC uses was reachable through the IP address displayed at boot. One thing to note when the VM is rebooted is that it relies on DHCP by default, meaning that the IP address can change between reboots. We resolved this issue by configuring a static IP address within the VM, making sure that the web interface is consistently available at the same address. 

\subsubsection{Standards and Framework support}
As a method and tool, MONARC is explicitly built around ISO/IEC 27005 as its primary methodological foundation. This is clear as it enforces the standard's procedure steps through its four stage treatment: context establishment, context modeling, risk evaluation and treatment, and implementation and monitoring. 

The platform's scoring approach, calculating risk based on threat likelihood, vulnerability level, and potential impact, is close to the risk estimation process that is described in ISO/IEC 27001, GDPR, and PCI--DSS through its built in risk models and reporting capabilities, thus making it suitable for enterprises pursuing certification or regulatory compliance. 

NIST SP--800 is not mentioned or supported by MONARC's methodology. Similarly, OCTAVE and FAIR are not directly supported. MONARC does share a similar asset-based approach with some of OCTAVE's focus on critical asset and organizational context, but it does not follow OCTAVE's structured, workshop-based assessment process. FAIR's probability and financial quantified risk modeling is also not reflected in MONARC's qualitative approach.

Overall, MONARC is best characterized as a structured, framework-driven risk analysis tool, with a strong direct alignment with IOS/IEC 27005. Its conformance to established standards is enforced, by MONARC rather than leaving it up to the user to figure it out themselves. This makes it suitable for organizations that seek a standardized and documented approach to information security risk management.

\subsection{Eramba}
\subsubsection{Tool Overview}
Eramba is a free open-source Governance, Risk, and Compliance (GRC) software that was developed by GRC professionals. The platform is web-based and uses GUI. All the functionality of the tool is accessible via a browser interface and is organised into modules. The tool does not require CLI when in normal use. Everything is done via the browser interface after installation.

Eramba is deployed as a self hosted web application and it is recommended to deploy it via Docker, although VMware is also possible. Eramba is available in two editions: a free community edition with no user or data limitations, and a paid Enterprise edition that adds more features and newer updates. The version used in this project is the free community edition.

Eramba is structured around a risk registere where risks are identified, assigned to owners/groups and then connected to relevant threats and vulnerabilites via a built in library that the user can customize. The risks that get put into the risk register go through 2 stages. The first stage is the initial analysis of the risk, and the second stage is the post treatment results. Each risk is calculated through likehlihood and the impact it has, these are configurable. Eramba also includes modules for compliance testing, asset managment, incident tracking and third party risks. The tool comes prearranged with a broad range of frameworks such as ISO 27005/27001, SOC2 and DORA. This will allow users to manage multiple frameworks at the same time.
